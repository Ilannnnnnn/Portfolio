JobAgent

Autonomous agentic pipeline for collecting, scoring and tracking job listings, with AI-powered CV tailoring.

In production

Context

Personal project developed in 2025–2026 during a permanent-position job search in AI engineering. Faced with the volume of listings to monitor daily across five different platforms, the goal was to fully automate the process: collection, deduplication, LLM-based scoring and CV tailoring — to apply to fewer positions, but better-targeted ones.

The agent runs locally and orchestrates five scrapers in parallel. Each new listing is scored by Claude Haiku on a 0-to-100 scale, benchmarked against the candidate profile. A Streamlit dashboard allows tracking applications and triggering CV tailoring in a single click.

Architecture — LangGraph pipeline

The core of the system is a LangGraph agentic graph composed of four nodes and one conditional edge. The five covered sources are Adzuna API, APEC via Apify, Indeed via JobSpy, Welcome to the Jungle via Apify and Google Jobs via JobSpy.

How it works

Step 1
Parallel collection — 5 scrapers in ThreadPoolExecutor, normalisation to a unified format, URL deduplication via a UNIQUE SQLite constraint. Deal-breakers filtered before insertion.
Step 2
Conditional scoring — if new listings exist, Claude Haiku analyses title + description + candidate profile. 1 s sleep between each call. Result validated by Pydantic.
Step 3
Dashboard — real-time database reads, filters by score / source / contract / status, detail view with salary-vs-market curve, Apply button.
Step 4
CV tailoring — Claude Sonnet reads the portfolio HTML (BeautifulSoup, 8,000 chars max / file), the YAML profile and the listing analysis. Streams to a tailored HTML file.
Step 5
Export — persistent listings.xlsx updated on each run, timestamped TXT report generated after every collection cycle.

Technical highlight

The should_score conditional edge — agents vs scripts

If zero new listings are detected after collection, the graph jumps directly past the score_batch node and moves to generate_report. This decision is made dynamically by LangGraph by reading state["new_offers_count"] — this is the fundamental difference between a sequential script and an agent: the execution flow is not fixed, it adapts to context at runtime.

In practice: the should_score() function returns either the string "score_batch" or "generate_report". LangGraph then routes the graph to the corresponding node. This conditional edge pattern is reusable for any branching logic — retry, fork, escalation.
