FitIA

Application mobile de coaching fitness personnalisé propulsée par l'IA — pipeline RAG sur 4M+ documents fitness, génération de plans d'entraînement structurés et suivi de progression.

Projet M2 · POC équipe

Contexte

Les applications de coaching fitness existantes proposent des programmes génériques, peu adaptés aux besoins individuels. L'objectif de FitIA était de construire un système capable de générer des conseils et des plans d'entraînement réellement personnalisés en combinant la puissance des LLMs avec une base de connaissances fitness structurée — le tout accessible depuis une application mobile native.

Développé en équipe dans le cadre du M2, FitIA est un Proof of Concept complet : une interface Flutter avec dark theme Material 3, un backend Python FastAPI, et un pipeline RAG (Retrieval-Augmented Generation) basé sur Weaviate indexant plus de 4 millions de documents fitness. L'application supporte deux modes de fonctionnement — standalone (appels Gemini directs depuis Flutter) et full RAG (FastAPI + Weaviate pour des réponses contextualisées).

Architecture

L'application repose sur une architecture à trois couches : l'interface Flutter communique via HTTP REST avec le serveur FastAPI, qui orchestre la recherche sémantique dans Weaviate et la génération de texte par Gemini. SentenceTransformers génère les embeddings 384 dimensions injectés dans la base vectorielle.

Pipeline RAG — Fonctionnement détaillé

Le pipeline RAG est le cœur du mode Full RAG : il transforme une question utilisateur en réponse contextualisée ancrée dans la base de connaissances fitness.

Étape 1
Encodage de la question — La question utilisateur est convertie en vecteur de 384 dimensions par SentenceTransformers (all-MiniLM-L6-v2), exécuté localement sur le serveur FastAPI.
Étape 2
Recherche sémantique near-vector — Le vecteur est comparé aux 4M+ documents indexés dans Weaviate via une recherche de similarité cosinus. Les top-K=5 documents les plus pertinents sont récupérés.
Étape 3
Formatage du contexte par type — Les documents récupérés sont formatés selon leur classe Weaviate : exercise, exercise_qa ou faq. Chaque type produit un bloc de contexte distinct.
Étape 4
Injection du profil utilisateur — Âge, IMC, objectifs, niveau et conditions médicales sont systématiquement injectés dans le prompt système. Le coach IA est ainsi conscient des contraintes et contre-indications de l'utilisateur.
Étape 5
Fenêtre d'historique — Les 10 derniers messages de la conversation sont inclus pour maintenir la cohérence du dialogue. L'IA peut ainsi faire référence aux échanges précédents et affiner ses recommandations.
Étape 6
Appel Gemini avec prompt enrichi — Le prompt final combine : prompt système coach fitness + contexte RAG + profil utilisateur + historique. Gemini génère une réponse Markdown (chat) ou JSON structuré (plan d'entraînement).
Étape 7
Retour à Flutter — La réponse est renvoyée à l'application mobile via FastAPI. Le Markdown est rendu nativement par flutter_markdown ; le JSON est parsé pour afficher le plan d'entraînement structuré.

Highlight technique

Génération JSON structurée via response_mime_type

Pour la génération du plan d'entraînement, l'API Gemini est appelée avec response_mime_type: "application/json". Cela force le modèle à retourner du JSON valide plutôt que du texte libre — sans post-processing ni regex fragile. Le schéma JSON (exercices, séries, reps, repos, conseils) est décrit dans le prompt système, et Gemini garantit la conformité structurelle de la réponse.

Ce pattern est combiné avec la conscience médicale : chaque prompt injecte les conditions de santé déclarées par l'utilisateur (hypertension, diabète, asthme…). Gemini adapte automatiquement les exercices proposés — intensités réduites, exercices à faible impact, avertissements intégrés dans les conseils techniques — sans logique de filtrage côté serveur.

Fine-tuning dataset via Claude Opus

La base de connaissances a été enrichie par un script de conversion automatique : les données brutes des datasets (exercices, FAQ) sont transformées en paires Q&A structurées via l'API Claude Opus (Anthropic SDK). Ce pipeline de fine-tuning a permis de générer des documents mieux adaptés à la recherche sémantique et à l'injection dans le contexte RAG.
